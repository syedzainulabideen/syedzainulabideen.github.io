SOLID - Interface Segregation Principle (ISP)

ISP states that clients ( classes or modules that use an protocol ) should not be forced to depend on protocols they do not use. This implies that we need to structure our protocols or interfaces in a manner that precisely fulfills the requirements of the entities that adhere to them.

Let’s understand by a quick example. Suppose you have multiple payment methods available in your codebase.

// Protocol for processing payments
protocol PaymentMethod {
    func processPayment(amount: Double)
		func refundPayment(amount: Double)
}

// Classes that implement the protocol

class CreditCardPayment: PaymentMethod {
    func processPayment(amount: Double) {
        print("Processing credit card payment of $\(amount)")
        // Actual payment processing logic for credit card
    }
    
    func refundPayment(amount: Double) {
        print("Refunding credit card payment of $\(amount)")
        // Actual refund processing logic for credit card
    }
}

class PayPalPayment: PaymentMethod {
    func processPayment(amount: Double) {
        print("Processing PayPal payment of $\(amount)")
        // Actual payment processing logic for PayPal
    }
    
	  func refundPayment(amount: Double) {
		    /*
					Imagine a scenario where there's no refund policy applicable to PayPal transactions. 
					Despite this, you are still required to implement the refundPayment function. 
					Alternatively, even if you make this function optional, client classes can still invoke the refundPayment function on instances of PayPalPayment.
				*/ 
		}
}

As you can observed in above example, PayPalPayment have to implement unnecessary function due to ISP violation. Let’s now refactor the above code to demonstrate how adhering to the ISP can result in a more maintainable and well-organized codebase.

// Protocol for processing payments
protocol PaymentMethod {
    func processPayment(amount: Double)
}

protocol RefundablePayment: PaymentMethod {
		func refundPayment(amount: Double)
}

// Classes that implement the protocol

class CreditCardPayment: RefundablePayment {
    func processPayment(amount: Double) {
        print("Processing credit card payment of $\(amount)")
        // Actual payment processing logic for credit card
    }
    
    func refundPayment(amount: Double) {
        print("Refunding credit card payment of $\(amount)")
        // Actual refund processing logic for credit card
    }
}

class PayPalPayment: PaymentMethod {
    func processPayment(amount: Double) {
        print("Processing PayPal payment of $\(amount)")
        // Actual payment processing logic for PayPal
    }
}

Now, The concrete classes, CreditCardPayment and PayPalPayment, demonstrate ISP adherence by conforming to the appropriate protocols. CreditCardPayment conforms to the RefundablePayment protocol, allowing it to implement both payment processing and refunding methods. But, PayPalPayment adheres to the PaymentMethod protocol, exclusively implementing payment processing as no refund functionality is required.

Another good example for ISP would be the Codable protocol in Swift Language. If you model class is supposed to be both Encoded and Decoded, you should use Codable. Else if model class is jsut to decode response data from network, you may want to conform to the Decodable protocol.


// The Product struct, conforming to the Decodable protocol, is specifically responsible for decoding response data.
struct Product: Decodable {
    let id: Int
    let name: String
    let price: Double
}

// Person struct adopting Codable, capable of decoding response data and encoding into data for transmission to the server.
struct Person: Codable {
    let name: String
    let age: Int
}

Key Points:

  • By segregating interfaces based on functionality and specialization, the code exemplifies modularity and clarity.
  • If a class requires multiple functionalities that are provided by different interfaces, you can use composition to combine these smaller interfaces rather than forcing the class to implement a large interface with unrelated methods.
  • It was introduced by Robert C. Martin as a guideline to help create more maintainable and flexible software systems.
