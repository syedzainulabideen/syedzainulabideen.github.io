Dependency Injection vs Dependency Inversion

Wednesday, May 17, 2023

Dependency Injection vs Dependency Inversion

Dependency injection and dependency inversion are two concepts used in software development to manage dependencies between different components of a system. While they are related, they have distinct purposes and approaches.

Dependency injection is a design pattern that allows components to be loosely coupled by providing their dependencies from external sources. In other words, instead of hardcoding dependencies within a component, the dependencies are passed to it from the outside. This promotes flexibility, modularity, and easier testing.

Example:

class Engine {
    func start() {
        print("Engine started!")
    }
}

class Car {
    let engine: Engine

    init(engine: Engine) {
        self.engine = engine
    }

    func start() {
        engine.start()
        print("Car started!")
    }
}

let engine = Engine()
let car = Car(engine: engine)
car.start()

In this example, the Car class depends on the Engine class to start. Instead of creating an Engine object within the Car class, the Engine instance is passed to the Car class through its initializer, demonstrating dependency injection.

On the other hand, Dependency inversion is a principle that guides the overall architecture of a system. It suggests that high-level modules should not depend on low-level modules directly, but both should depend on abstractions. This principle helps to decouple modules and make them more interchangeable and reusable.

Example:

protocol Engine {
    func start()
}

class ElectricEngine: Engine {
    func start() {
        print("Electric engine started!")
    }
}

class GasolineEngine: Engine {
    func start() {
        print("Gasoline engine started!")
    }
}

class Car {
    let engine: Engine

    init(engine: Engine) {
        self.engine = engine
    }

    func start() {
        engine.start()
        print("Car started!")
    }
}

let electricEngine = ElectricEngine()
let gasolineEngine = GasolineEngine()

let electricCar = Car(engine: electricEngine)
electricCar.start()

let gasolineCar = Car(engine: gasolineEngine)
gasolineCar.start()

In this example, we have defined an Engine protocol that represents the abstraction for an engine. Both ElectricEngine and GasolineEngine conform to this protocol, representing the concrete implementations of different types of engines.

The Car class depends on the Engine protocol instead of the concrete engine types directly. This allows us to easily switch between different types of engines without modifying the Car class. By adhering to the dependency inversion principle, the high-level Car class depends on the abstraction (Engine protocol), while the concrete engine implementations depend on that abstraction as well. This decouples the Car class from specific engine implementations, making it more flexible and promoting reusability.
