SOLID - Open Close Principle (OCP)

OCP states that software entities (such as classes, modules, functions, etc.) should be open for extension but closed for modification. In simple words it means that once a entity is created and working as intended, its internal code should not be modified to add new features or behaviors.

So how can then we add features or update the functionalities? And the answer is by extending the existing entities with by Subclassing, Extensions or Protocols/Interfaces etc.

This principle aims to reduce the risk of introducing errors and regression issues by preventing unnecessary modifications to already tested and working code.

class Shape {
    // Common properties and methods for all shapes
}

class Circle: Shape {
    func draw() {
        // Code to draw a circle
    }
}

class Square: Shape {
    func draw() {
        // Code to draw a square
    }
}

class ShapeRenderer {
    func render(shape: Shape) {
        shape.draw()
    }
}

In this example, the ShapeRenderer class adheres to the Open-Closed Principle. It’s open for extension because you can easily introduce new shapes (e.g., a triangle) without modifying the existing code. You can create a new class for the triangle shape that inherits from Shape and implements its own draw method.

Below is another example using the Protocol to present how OCP works.

protocol Payable {
    func makePayment(amount: Double)
}

struct CreditCard: Payable {
    func makePayment(amount: Double) {
        print("Paying $\(amount) via Credit Card")
    }
}

struct Paypal: Payable {
    func makePayment(amount: Double) {
        print("Paying $\(amount) via Paypal")
    }
}

class PaymentProcessor {
    func processPayment(paymentMethod: Payable, amount: Double) {
        paymentMethod.makePayment(amount: amount)
    }
}

We have a protocol Payable that defines the contract for making payments. There are two structs, CreditCard and Paypal, both conforming to the Payable protocol. They implement the makePayment method differently based on the payment method.

The PaymentProcessor class has a method processPayment that accepts a Payable object and an amount. It calls the makePayment method on the provided payment method object. So now if we need to add a new Payment method in our flow, we don’t need to modify the PaymentProcessor or Other payment methods. We just need to add new Payment method that conforms to the protocol Payable

Key Points:

  • By not modifying existing code, you minimize the chances of introducing bugs and unintended side effects.
  • The principle encourages the creation of well-defined abstractions, making it easier to add new features or behaviors by creating new classes that adhere to existing flow.
  • When existing code is open for extension, new features can be added to different parts of the application without duplicating code. This promotes code reuse and reduces redundancy.
